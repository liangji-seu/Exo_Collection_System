# AK80-9 右电机位置反馈不更新问题诊断

## 1. 问题摘要

上位机已经能够通过 Teensy USB 串口稳定接收两台 AK80-9 电机的状态帧，但实际转动右电机输出轴时出现以下现象：

- 右电机速度会随转动发生明显变化；
- 右电机位置始终保持同一个数值，不随转动变化；
- 因此 Collector 中右电机位置曲线表现为一条水平直线；
- 左电机未转动时，位置、速度和扭矩均保持不变；
- 当前电机处于 `disabled` 状态，所做测试全部为只读测试，上位机没有发送使能、力矩或位置控制命令。

这说明 USB 串口、35 字节状态帧解析、采样循环和 UI 刷新均在正常工作。问题已经缩小到 Teensy 接收到电机 CAN 反馈后，对“位置字段”的识别、解析或更新环节。

## 2. 当前软硬件配置

- 电机：CubeMars AK80-9 KV100，固件 AK80_9_SE_V3.2
- 左电机 CAN ID：65，即 `0x41`
- 右电机 CAN ID：33，即 `0x21`
- CAN：1 Mbps，扩展帧，周期反馈模式
- 电机反馈频率：约 50 Hz
- Teensy 状态帧输出频率：约 200 Hz
- Teensy USB VID/PID：`16C0:0483`
- 当前测试串口：自动识别为 COM12（软件没有硬编码 COM12）
- 右电机反馈扩展 CAN ID：`0x2921`，即 `(0x29 << 8) | 0x21`
- 左电机反馈扩展 CAN ID：`0x2941`，即 `(0x29 << 8) | 0x41`

## 3. 上位机串口状态帧协议

Teensy 向上位机发送固定 35 字节状态帧：

| 偏移 | 长度 | 字段 | 类型/单位 |
|---:|---:|---|---|
| 0 | 1 | head | `0xCC` |
| 1 | 2 | seq | uint16 little-endian |
| 3 | 1 | state | `0=disabled`，`2=running` |
| 4 | 1 | error_code | 状态/故障码 |
| 5 | 4 | left_pos | float32，rad |
| 9 | 4 | left_vel | float32，rad/s |
| 13 | 4 | left_torque | float32，N·m |
| 17 | 4 | right_pos | float32，rad |
| 21 | 4 | right_vel | float32，rad/s |
| 25 | 4 | right_torque | float32，N·m |
| 29 | 4 | teensy_time_us | uint32 |
| 33 | 1 | crc8 | CRC-8 |
| 34 | 1 | tail | `0x55` |

上位机已按该协议正确解析。真实串口测试中状态帧 CRC、帧头、帧尾和时间戳均正常。

## 4. 两次真实只读测试结果

### 4.1 电机未移动时

连续读取约 3 秒：

- 收到 600 个样本，约 200 Hz；
- CRC/格式错误：0；
- 时间戳丢帧：0；
- 状态：`disabled`；
- 错误状态：正常；
- 左位置：始终 `1.548107147 rad`；
- 左速度：始终 `0 rad/s`；
- 左扭矩：始终 `0 N·m`；
- 右位置：始终 `55.850536346 rad`；
- 右速度：始终 `0 rad/s`；
- 右扭矩：始终 `0 N·m`。

### 4.2 人工转动右电机时

在连续 6 秒只读采样期间人工转动右电机，结果如下：

| 字段 | 首值 | 末值 | 最小值 | 最大值 | 极差 | 不同值数量 |
|---|---:|---:|---:|---:|---:|---:|
| left_position | 1.548107 | 1.548107 | 1.548107 | 1.548107 | 0 | 1 |
| left_velocity | 0 | 0 | 0 | 0 | 0 | 1 |
| left_torque | 0 | 0 | 0 | 0 | 0 | 1 |
| right_position | 55.850536 | 55.850536 | 55.850536 | 55.850536 | 0 | 1 |
| right_velocity | 0 | -3.623636 | -4.005946 | 4.432582 | 8.438528 | 101 |
| right_torque | 0 | 0 | 0 | 0 | 0 | 1 |

总样本数为 1200。右电机速度明确检测到了正反方向运动，但右电机位置在全部 1200 帧中完全相同。

## 5. 已经排除的问题

以下环节基本可以排除：

1. **USB 串口未收到数据**：实际稳定收到约 200 帧/秒。
2. **状态帧错位或 CRC 错误**：CRC/格式错误为 0，其他字段均能正确读取。
3. **左右电机通道映射错误**：人工移动右电机时只有 `right_velocity` 明显变化。
4. **Collector 没有刷新曲线**：直接绕过 UI，从 Adapter 读取数组，得到的 `right_position` 本身就是常数。
5. **采样频率不足**：采样为 200 Hz，右速度变化被完整观察到。
6. **上位机向电机发送了控制命令**：诊断脚本只调用读取生命周期，没有串口写入操作。
7. **仅仅因为电机处于禁用状态而无法获得任何反馈**：虽然状态为 `disabled`，但右速度仍随人工转动发生变化，说明 CAN 周期反馈仍在到达和更新。

## 6. 当前 Teensy 位置解析代码

当前固件在 `parseMotorFeedback()` 中这样解析电机反馈：

```cpp
const int16_t positionRaw =
    (int16_t)(((uint16_t)msg.buf[0] << 8) | msg.buf[1]);
const int16_t speedRaw =
    (int16_t)(((uint16_t)msg.buf[2] << 8) | msg.buf[3]);
const int16_t iqRaw =
    (int16_t)(((uint16_t)msg.buf[4] << 8) | msg.buf[5]);

motor->pos_rad =
    (float)positionRaw * 0.1f * MotorParams::RAD_PER_DEGREE;
motor->vel_rad_s =
    (float)speedRaw * 10.0f * MotorParams::ERPM_TO_OUTPUT_RAD_S;
motor->torque_nm =
    (float)iqRaw * 0.01f * MotorParams::TORQUE_CONSTANT_NM_PER_A;
```

CAN 帧过滤条件为：

```cpp
if (!msg.flags.extended ||
    msg.len != 8 ||
    (msg.id & 0xFFFFFF00UL) != (0x29UL << 8)) {
    return;
}
```

随后 Teensy 将 `motor->pos_rad`、`motor->vel_rad_s` 和 `motor->torque_nm` 原样打包成 float32 状态帧发送给上位机。

## 7. 最可能的问题原因

目前最可疑的是对 AK80-9 V3.2 周期反馈帧 `0x29xx` 的前两个数据字节解释不正确。可能包括：

1. `msg.buf[0:2]` 并不是当前配置下的位置字段；
2. 位置字段不是当前假设的 big-endian `int16_t`；
3. 位置字段可能使用其他长度、符号或缩放规则；
4. 当前收到的是另一种 `0x29` 反馈子格式，速度字段位置碰巧正确而位置字段含义不同；
5. 电机周期回报配置没有启用实时位置，前两个字节返回的是固定参数或固定累计值；
6. 固件需要根据 AK80_9_SE_V3.2 实际使用的协议版本重新核对反馈格式。

其中，当前右位置 `55.850536 rad` 约等于 `3200°`。该值明显超出代码中用于 MIT 控制的 `±12.56 rad` 范围，是位置解释或缩放异常的重要证据。它也恰好接近：

```text
32000 count × 0.1 degree/count = 3200 degree ≈ 55.85 rad
```

这说明 `msg.buf[0:2]` 很可能长期组成了约 32000 的原始数，但该数未必是实际输出轴位置。

## 8. 建议的下一步诊断

应优先在 Teensy 固件中临时记录右电机原始 CAN 帧，而不是继续修改上位机。

### 8.1 抓取转动前后的原始 CAN 数据

对 ID `0x2921` 输出以下内容：

- 完整扩展 CAN ID；
- `msg.len`；
- `msg.buf[0]` 到 `msg.buf[7]` 的十六进制值；
- 同一帧按当前算法解析出的 `positionRaw`、`speedRaw`、`iqRaw`；
- 抓取静止、正向转动、反向转动三个阶段。

注意：不要把调试文本混入当前二进制 USB 状态帧使用的同一个串口，否则会破坏上位机的 35 字节帧解析。可选择：

- 临时使用另一个硬件串口输出调试信息；
- 暂时使用只打印原始 CAN 的专用诊断固件；
- 或把原始 8 字节 CAN 数据加入一个新的、明确分帧并带 CRC 的诊断帧。

### 8.2 判断位置实际位于哪些字节

人工缓慢转动右电机时逐字节比较：

- 哪些字节持续变化；
- 哪些字节随转动方向改变符号或趋势；
- 哪些字节与速度变化对应；
- 前两个字节是否真的完全不变。

如果 `buf[0:2]` 在原始 CAN 中也不变，则问题是电机反馈配置或协议字段理解；如果 `buf[0:2]` 会变化而 Teensy 的 `right_pos` 不变，则问题是 Teensy 解析或变量更新逻辑。

### 8.3 对照正确版本的官方协议

必须使用与以下条件完全对应的 CubeMars 协议说明：

- AK80-9 KV100；
- AK80_9_SE_V3.2；
- V3.0 扩展 CAN 协议；
- 周期回报模式；
- 功能码 `0x29`；
- 当前电机参数工具实际写入的反馈内容配置。

不要直接套用 AK60-6 V1.1、传统 MIT 8 字节反馈或其他固件版本的数据布局。

## 9. 修复后的验收标准

固件修复后至少应满足：

1. 电机静止时位置稳定，速度接近 0；
2. 人工正向转动右电机时，右位置连续单调变化，右速度方向正确；
3. 反向转动时，位置反向变化，速度符号反转；
4. 左右电机独立转动时，只更新对应一侧数据；
5. 位置数值和机械实际转角量级一致；
6. Teensy 状态流仍维持约 200 Hz；
7. CRC/格式错误为 0；
8. 不移动时连续四个左右状态帧出现重复是允许的，因为电机 CAN 反馈约为 50 Hz，而 Teensy 状态帧为 200 Hz；
9. Collector 和最终 HDF5 中的六个通道均应保留：左右位置、速度和估算扭矩。

## 10. 结论

上位机目前能够正确、稳定地接收和解析 Teensy 35 字节状态帧。人工移动右电机时，`right_velocity` 的变化证明右电机 CAN 反馈确实到达 Teensy，并被传输到上位机；但 `right_position` 在全部样本中严格不变，且固定值约 `55.85 rad` 明显可疑。

因此当前首要修复目标不是 Collector UI 或 Python 串口解析，而是核实并修正 Teensy 对 AK80-9 V3.2 `0x29xx` 周期反馈帧中位置字段的解析与缩放。
